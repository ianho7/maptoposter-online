# 代码审查

审查目标：$ARGUMENTS

请先读取技能文档作为审查基准：

<bash>
cat .claude/skills/react-19/SKILL.md
</bash>

然后确定审查范围：

- 若 `$ARGUMENTS` 为文件路径 → 审查该文件
- 若 `$ARGUMENTS` 为目录路径 → 递归审查其中所有 `.ts` / `.tsx` 文件
- 若 `$ARGUMENTS` 为空 → 审查整个 `src/` 目录

---

## 审查清单

### 1. React 19 规范（以读取的 SKILL.md 为准）

- **禁止手动 memoization**：标记所有 `useMemo`、`useCallback`、`React.memo` 用法 —— React Compiler 自动处理优化
- **导入方式**：必须使用具名导入 `import { useState } from "react"`，禁止默认导入或命名空间导入
- **ref 作为 prop**：React 19 中 `forwardRef` 已废弃，ref 应作为普通 prop 传递
- **`use()` hook**：条件性读取 Context 或 Promise 时，优先使用 `use()` 而非 `useContext`
- **`useActionState`**：表单提交 + pending 状态场景，优先使用 `useActionState`
- **Server / Client 边界**：检查 `"use client"` 是否必要；不依赖浏览器 API 的组件默认应为 Server Component

### 2. TypeScript 质量

- **禁止裸 `any`**：每处 `any` 必须有注释说明原因，或替换为准确类型 / `unknown`
- **空值安全**：检查未守护的 `.value`、数组越界访问、遗漏的可选链
- **类型推断**：避免冗余类型注解；同样标记推断不足的地方
- **`as` 断言**：标记不安全强转，优先使用类型守卫或 `satisfies`
- **枚举**：优先使用 `const` 对象 + `as const` + 联合类型，而非 `enum`
- **泛型约束**：检查泛型函数的约束是否合理，避免过于宽泛

### 3. Tailwind CSS v4 用法

- **避免滥用任意值**：颜色、间距、字号应使用设计 token，而非 `text-[13px]`、`bg-[#abc]`
- **移动优先**：验证断点使用是否遵循 `sm:` / `md:` 等移动优先顺序
- **暗色模式**：若项目启用暗色模式，检查 `dark:` 变体是否一致覆盖
- **禁用内联 style 做布局**：可用 Tailwind 表达的布局不应使用 `style={{}}`
- **条件类合并**：使用 `clsx` / `cn` / `twMerge` 时，检查条件类是否统一走合并函数

### 4. Vite 构建相关

- **动态导入**：路由级页面或大型组件应使用 `React.lazy` + `Suspense` 或 `import()` 做代码分割
- **环境变量**：只能通过 `import.meta.env.VITE_*` 访问环境变量，标记 `process.env` 用法
- **静态资源**：图片、SVG 等应通过 `import` 引入，而非魔法字符串路径
- **`vite.config.ts`**（在范围内时）：检查插件冗余、`resolve.alias` 正确性、`build.target` 合理性

### 5. 通用代码质量

- **组件体积**：超过约 200 行的组件建议拆分
- **单一职责**：组件 / hook 应只做一件事，标记"上帝组件"
- **自定义 hook 提取**：含 2 个以上 state 或副作用的逻辑应提取为 `use*` hook
- **错误边界**：异步数据获取组件应包裹 Error Boundary
- **可访问性**：检查缺失的 `aria-*`、未标注的表单项、非语义化 HTML（如 `<div onClick>`）
- **列表 key**：动态列表中禁止使用数组下标作为 `key`
- **调试代码**：标记生产代码路径中残留的 `console.log` / `console.error`
